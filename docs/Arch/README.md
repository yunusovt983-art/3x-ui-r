# Архитектура 3x-ui сквозь призму DDD

> Цель документа — **извлечь полезные идеи** из исходного кода [3x-ui](https://github.com/MHSanaei/3x-ui)
> (Go, версия `3.3.1`) и **спроектировать систему с нуля на Rust**, опираясь на принципы
> и философию Domain-Driven Design.

Это не описание «как устроен Go-код построчно», а **реконструкция предметной области**:
какие сущности, инварианты и сценарии реально живут внутри панели, как они группируются в
ограниченные контексты, и как эту модель чисто перенести на Rust.

## Что это за система (в одном абзаце)

3x-ui — это **веб-панель управления прокси-ядром Xray-core**. Она не передаёт трафик сама:
она конфигурирует и держит под контролем процесс `xray`, заводит пользователей (клиентов) на
входящих соединениях (inbounds), считает их трафик, ограничивает по объёму / сроку / числу IP,
отдаёт клиентам ссылки-подписки (vless/vmess/trojan/ss/hysteria/clash/json), federation-режимом
объединяет несколько панелей-нод, и общается с админом через Web-UI и Telegram-бота.

## Ключевой вывод проекта (TL;DR)

1. **Домен Xray-core стоит держать как отдельный ограниченный контекст с Anti-Corruption Layer.**
   В Go он размазан между `internal/xray` и `internal/web/service`. На Rust его надо изолировать
   за портом `XrayController`, чтобы остальная система не зависела от формата JSON-конфига Xray.
2. **«Горячее» применение конфигурации (hot-diff) — это центральная доменная идея**, а не
   техническая деталь. Она заслуживает явной модели (см. [04](3x-ui/04-domain-model.md#hot-diff)).
3. **На Rust не нужно переписывать прокси-ядро.** Панель остаётся оркестратором. Xray-core
   (или альтернативное Rust-ядро вроде [`shoes`](07-rust-ecosystem-survey.md)) — это внешний
   движок за ACL.
4. **Слой приложения и домен должны быть синхронными и чистыми; вся async-ткань (БД, gRPC, HTTP,
   процессы) — в адаптерах.** Это прямой перенос гексагональной архитектуры в идиоматичный Rust.
