# 06 — Паттерны и что переносим

## 6.1. Архитектурные паттерны shoes

| Паттерн | Где | Суть |
|---------|-----|------|
| **Ports & Adapters (внутри процесса)** | все межмодульные связи | Контексты общаются через trait'ы (`TcpServerHandler`, `Resolver`, `ProxyConnector`), реализации подставляют фабрики из конфига. |
| **Декоратор транспортов** | `crypto/*`, `vless/vision_stream.rs` | TLS оборачивает TCP, Vision оборачивает TLS — вложенные обёртки над `AsyncStream`. |
| **Фабрика по sum-type** | `tcp/*_handler_factory.rs` | `ServerProxyConfig` (enum) → `Box<dyn Handler>`. Добавить протокол = вариант + ветка фабрики. |
| **Результат как алгебраический тип** | `TcpServerSetupResult` | Handler возвращает *намерение* (`TcpForward`/`Udp`/`AlreadyHandled`), а не выполняет его — диспетчер решает. |
| **Ленивое значение с кэшем** | `ResolvedLocation` | Имя резолвится один раз, кэшируется в VO, переиспользуется по цепочке. |
| **Композиция резолверов** | `resolver.rs`, `dns/*` | `Timeout(Refreshing(Composite([...])))` — поведение собирается обёртками. |
| **Атомарный hot-reload** | `main.rs:316` | Изменение файла → `abort()` всех задач → пауза → полный перезапуск. Просто и без частичных состояний. |
| **Round-robin через atomics** | `client_proxy_chain.rs` | Балансировка per-hop через `AtomicU32::fetch_add`, без блокировок и координации. |
| **Топологическое разрешение групп** | `config/validate.rs` | Ссылки на именованные группы разворачиваются по Kahn с детектом циклов на этапе валидации. |
| **Default-deny маршрутизация** | `client_proxy_selector.rs` | Нет совпадения правила → Block. Безопасно по умолчанию. |
| **Constant-time auth** | `naiveproxy/user_lookup.rs` | BLAKE3-hash → O(1) lookup + `subtle::ConstantTimeEq`. |

## 6.2. Что переносим в наш проект

Связано с [`../09-ideas-from-rust-projects.md`](../09-ideas-from-rust-projects.md) и
[`../06-rust-redesign.md`](../06-rust-redesign.md). Помним правило clean-room из
[`../08-licensing-compliance.md`](../08-licensing-compliance.md): из shoes (MIT) **можно и код** — но
только в пути B; в пути A берём паттерны.

### Сразу применимо (путь A — панель)
1. **Типизированная модель конфига inbound через sum-types + `deny_unknown_fields`.** Прямой ориентир —
   `ServerProxyConfig`/`ServerConfig`. Невыразимая невалидная комбинация полей. Наш ACL к Xray сериализует
   эти типы в его JSON.
2. **Эргономичные serde-helpers** (`NoneOrSome`/`OneOrSome`/`ConfigSelection`) — для удобного и строгого
   формата конфига панели и именованных ссылок («define once, reference everywhere» = наши шаблоны inbound).
3. **Топологическое разрешение именованных групп с детектом циклов** — если в панели будут группы/шаблоны/
   наследование настроек.
4. **Default-deny + первое-правило-выигрывает** — модель для нашего Routing-контекста (валидация порядка правил).
5. **Модель «результат = намерение»** (`TcpServerSetupResult`) — стиль для команд нашего домена: метод
   возвращает решение/событие, исполнение — в адаптере.

### Ориентир для пути B (своё ядро)
6. **Trait-карта data-plane готова:** `AsyncStream` + `TcpServerHandler`/`TcpClientHandler` +
   `Socket/ProxyConnector` + `Resolver` — это и есть порты прокси-ядра. Можно взять архитектуру (и, под
   MIT, код) как фундамент.
7. **Декоратор-транспорты** — как композировать TLS/REALITY/Vision/WS/QUIC поверх одного `AsyncStream`.
8. **REALITY/Vision/Hysteria2/TUIC уже реализованы** — самый дорогой и рискованный кусок крипто/обфускации
   не пришлось бы писать с нуля.
9. **UDP-мультиплексор и smoltcp-TUN** — готовые решения для UDP-ассоциаций и VPN-режима.

### Чего в shoes НЕТ — и это ровно наша ценность (control-plane)
- ❌ Управляющий API (add/remove user на лету) — у shoes только hot-reload файла.
- ❌ Per-user durable учёт трафика, лимиты по объёму/сроку/IP, online-tracking.
- ❌ Подписки, Web-UI, Telegram, multi-node federation, БД пользователей.

→ Подтверждает стратегию: **наша ценность — control-plane**, который shoes (как и xray) принципиально не
покрывает. Путь A (панель + готовое ядро) остаётся рекомендованным; shoes — реалистичный «план B» для
движка, если когда-нибудь уберём Go-зависимость.

## 6.3. Чему shoes учит про идиоматичный Rust здесь
- Полиморфизм доменных вариантов — **enum + trait-объекты**, а не наследование.
- I/O-границы — **trait'ы-порты**; конкретика собирается из конфига фабриками (тот же гексагон, что в нашем `06`).
- Состояние под конкурентность — **atomics вместо локов** там, где достаточно (round-robin, activity).
- Конфиг — **источник истины**; валидация и разворачивание ссылок — на старте, рантайм работает с уже
  проверенным графом объектов.
